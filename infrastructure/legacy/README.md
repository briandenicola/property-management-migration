# Legacy Infrastructure — Windows Server 2016 VM + Active Directory

This Terraform configuration provisions the full legacy environment for the PropertyManager application,
including a Domain Controller, IIS application server, and a Windows client workstation -- all
domain-joined to simulate a typical enterprise on-premises setup.

## What It Creates

| Resource | Purpose |
|----------|---------|
| Resource Group (auto-generated) | Container for all legacy resources |
| Virtual Network + Subnet | Isolated network with auto-generated CIDR blocks |
| **Domain Controller VM** (Standard_B2s) | AD DS + DNS for `bjdazure.tech` domain |
| **IIS/SQL Server VM** (Standard_B2ms) | Hosts the PropertyManager app (domain-joined) |
| **Windows Client VM** (Standard_B2s) | Domain-joined workstation for testing |
| Public IPs (Static) | One per VM for RDP access |
| NSG | Firewall rules: HTTP (80), HTTPS (443), RDP (3389 from your IP only) |
| Data Disk (64 GB) | Dedicated SQL Server data file storage on IIS VM |
| Custom Script Extensions | Automated DC promotion, domain joins, IIS + SQL setup |

## Provisioning Order

Terraform handles dependencies automatically:
1. **VNet + Subnet + NSG** -- networking foundation
2. **Domain Controller VM** -- deployed and promoted to DC (creates the forest)
3. **VNet DNS updated** -- points to DC's private IP
4. **IIS VM + Client VM** -- deployed and domain-joined after DNS is ready

## Connecting to VMs

All VMs are private (no public IPs). Use **Azure Bastion** to connect:

1. Open the [Azure Portal](https://portal.azure.com)
2. Navigate to the VM you want to access
3. Click **Connect** > **Bastion**
4. Enter credentials:
   - **Before domain join:** `pmadmin` / (password from `terraform output -raw vm_admin_password`)
   - **After domain join:** `alice.manager@bjdazure.tech` / `P@ssw0rd2024!`

Or via Azure CLI:
```bash
# Get the Bastion and VM names
BASTION=$(terraform output -raw bastion_name)
RG=$(terraform output -raw resource_group_name)

# Connect (opens native RDP client)
az network bastion rdp --name $BASTION --resource-group $RG --target-resource-id <vm-resource-id>
```

From the repository root:

```bash
# 1. Preview what will be created
task legacy:plan

# 2. Provision the VM
task legacy:up

# 3. Get your credentials and connection info
task legacy:creds

# 4. Connect via RDP or browse the site
task legacy:browse
```

## Manual Terraform Usage

If you prefer to run Terraform directly:

```bash
# 1. Initialize Terraform
terraform init

# 2. Review the plan (default region: canadacentral)
terraform plan -var "location=canadacentral"

# 3. Apply
terraform apply -var "location=canadacentral"

# 4. View outputs
terraform output resource_group_name
terraform output -raw vm_public_ip
terraform output -raw rdp_connection_string
terraform output -raw vm_admin_password
terraform output -raw sql_sa_password
terraform output -raw iis_url
```

## Configuration

### Terraform Variables

| Variable | Type | Default | Notes |
|----------|------|---------|-------|
| `location` | string | -- | Azure region (required, pass via CLI: `-var "location=canadacentral"`) |
| `tags` | string | "Property Manager" | Application identifier tag |
| `admin_username` | string | "pmadmin" | VM administrator username |
| `domain_name` | string | "bjdazure.tech" | Active Directory domain FQDN |

**No terraform.tfvars file is needed.** All sensitive values (VM admin password, SQL SA password) are randomly generated by Terraform.

### Auto-Generated Resources

Resource naming uses randomized names to avoid conflicts:
- Resource Group: `{pet}-{id}_rg` (e.g., `dolphin-a1b2_rg`)
- VM: `{pet}-{id}-vm`
- VNet: `{pet}-{id}-vnet`
- Subnet: `{pet}-{id}-subnet`
- NIC: `{pet}-{id}-nic`
- NSG: `{pet}-{id}-nsg`
- Public IP: `{pet}-{id}-pip`

**Admin password & SQL SA password** are randomly generated (16 chars, include special characters).

## Post-Provisioning

The Custom Script Extension runs during VM creation and:
1. Initializes and formats the data disk (F: drive)
2. Installs IIS with ASP.NET 4.6 features
3. Downloads and installs SQL Server 2016 Express
4. Creates the `PropertyManagement` database on the data disk
5. Configures the IIS site pointing to `C:\inetpub\wwwroot\PropertyManagement`

## Security Notes

- **No public IPs on VMs** -- all access via Azure Bastion only
- **Bastion** is the only resource with a public IP (required by Azure)
- **Passwords:** All sensitive values are randomly generated and marked `sensitive` in Terraform outputs -- never stored in tfvars
- **Never commit Terraform state files** to version control (they contain sensitive data)
- All VMs are domain-joined to `bjdazure.tech` (configurable via `domain_name` variable)
- NSG allows HTTP/HTTPS only within the VNet (no external inbound)

## Active Directory Configuration

After provisioning, the DC is automatically promoted. To create the demo OUs, groups, and users, RDP into the DC and run:

```powershell
# Run on the Domain Controller after reboot
C:\configure-ad-objects.ps1 -DomainName "bjdazure.tech"
```

This creates:
- **OU:** PropertyManager/Users, PropertyManager/Groups
- **Groups:** PropertyManager-Admins, PropertyManager-Users
- **Demo Users:**

## Testing Windows Authentication

1. RDP into the **client VM** as a domain user (e.g., `alice.manager@bjdazure.tech`)
2. Open a browser and navigate to the IIS VM's IP (or hostname)
3. Windows Auth is transparent -- the browser automatically sends credentials
4. Admin users see full CRUD; regular users see read-only for properties/tenants
